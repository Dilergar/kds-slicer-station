# Инструкция по подключению модуля KDS Slicer Station

> Гайд для IT-команды заказчика. Описывает как добавить модуль нарезчика
> в существующую продакшн-систему с уже работающей основной KDS и рабочей
> БД `arclient` (PostgreSQL 12+).
>
> **Исходная точка.** У вас уже развёрнут и работает проект KDS: продакшн-БД
> `arclient`, касса пишет заказы, основная KDS отображает их поварам. Этот
> гайд добавляет **рядом** станцию нарезчика — отдельный UI и backend,
> который читает те же заказы, но показывает их под углом нарезчика.
>
> **Гарантия неприкосновенности вашей БД.** Модуль **не изменяет** структуру
> существующих таблиц (`docm2_*`, `ctlg*`, `rgst*`). Все свои данные он
> хранит в новых таблицах с префиксом `slicer_*`, которые создаются
> миграциями из этого архива.
>
> **В чужие таблицы модуль по умолчанию ничего не пишет.** В частности,
> `docm2tabl1_items.docm2tabl1_cooked` **не трогается**: это поле остаётся
> под управлением основной KDS (раздача/пасс отмечают готовность всего
> блюда). Нарезчик закрывает только свою часть — этот факт сохраняется в
> нашей теневой таблице `slicer_order_state` и **не виден** вашим
> дашбордам раздачи, повара или мобильному приложению.
>
> Единственная опциональная точка записи в чужую таблицу —
> `rgst3_dishstoplist` через раздел 10 инструкции. Она **выключена по
> умолчанию** и активируется только по вашему решению.
>
> **Плюс BEFORE DELETE триггер** на `rgst3_dishstoplist` (миграция 011) —
> читает OLD row при удалении и копирует в нашу `slicer_stop_history`.
> НЕ изменяет схему чужой таблицы, НЕ блокирует DELETE, НЕ пишет ничего
> в чужие таблицы. Нужен чтобы аналитика стопов работала полноценно —
> без него отчёт не увидит стопы, которые кассир ставил и снимал в
> течение одной смены (rgst3 удаляет row безвозвратно).

## 1. Предварительные требования

### Софт на машине, где будет крутиться модуль

| Компонент | Минимум | Рекомендуется | Как проверить |
|---|---|---|---|
| Node.js | 20.x | **22.x LTS** | `node --version` |
| npm | 10.x | 10.9+ | `npm --version` |
| PostgreSQL | 12+ | **18** | `psql --version` |
| ОС | любая | Windows / Linux / macOS | — |

Модуль протестирован на PostgreSQL 18.3 с БД, восстановленной из дампа
PG 9.6 — обратная совместимость работает.

### Доступ к БД `arclient`

Модуль подключается к **той же** БД, с которой работает основная KDS —
никакой отдельной базы не требуется. Пользователь PostgreSQL, которого
вы укажете в `.env`, должен иметь:

- `CONNECT` на БД `arclient`
- `SELECT` на все таблицы `docm2_*`, `ctlg*`, `rgst3_dishstoplist`
- Полный `CRUD` на новые таблицы `slicer_*` (создаются миграциями модуля)
- **Никаких прав на запись в ваши `docm2_*` / `ctlg*` таблицы модулю не нужно.**
  Если используете раздел 10 (двусторонний стоп-лист) — тогда дополнительно
  `INSERT`/`DELETE` на `rgst3_dishstoplist`, но только там.

Имя БД (`arclient`) не хардкодится нигде в коде, читается из `server/.env`.
Если у вас БД называется иначе — укажите настоящее имя в `DB_NAME` на шаге 5.

### Порты

| Порт | Кто слушает | Можно ли изменить |
|---|---|---|
| `3001` | backend модуля (Express API) | Да, через `SERVER_PORT` в `.env`; тогда же обновите `vite.config.ts` |
| `3000` | frontend dev-сервер (Vite) | Да, через `vite.config.ts`; в проде фронт отдаётся статикой и этот порт не нужен |

Основная KDS и касса никаких портов не делят с модулем.

## 2. Размещение проекта на сервере

Распакуйте архив модуля в любое удобное место на сервере, например:

- Linux: `/opt/kds-slicer-station/`
- Windows: `C:\kds-slicer-station\`

После распаковки вы должны увидеть такую структуру:

```
kds-slicer-station/
├── server/                # backend (Express + PostgreSQL)
│   ├── migrations/        # SQL-миграции модуля (001–019)
│   ├── src/               # код backend
│   └── package.json
├── src/... (components, hooks, services, ...)  # frontend
├── index.html
├── package.json           # frontend-зависимости
├── vite.config.ts
├── Инструкция.md          # этот файл
├── CLAUDE.md              # внутренняя документация для разработчика модуля
├── README.md
└── BD_docs/               # документация схемы БД модуля
```

Дальше все команды из инструкции предполагают, что **текущая директория —
корень проекта** (там где лежит `package.json` и `Инструкция.md`), если
явно не сказано иначе.

## 3. Настройка UUID цеха кухни (критический шаг)

**⚠️ Выполните этот шаг ДО миграций и ДО первого запуска.**

Модуль показывает нарезчику только заказы с позициями, привязанными к
складу «Кухня». Фильтр настроен по UUID, который **захардкожен в коде
под конкретный ресторан** (того, для кого делался модуль). Если не
обновить под ваш ресторан — модуль будет работать, UI откроется, ошибок
в логах не будет, но **доска KDS окажется полностью пустой**.

Это самая частая и самая молчаливая ошибка при переносе модуля.

### Шаг 3.1. Узнать UUID вашего цеха «Кухня»

Выполните запрос к вашей БД `arclient`:

```sql
SELECT suuid, name FROM ctlg17_storages ORDER BY name;
```

Пример вывода (у вас UUID будут свои):

```
                suuid                 | name
--------------------------------------+-------
 b54a3a11-7e05-403d-ac70-34ed7c51c93e | Бар
 123fa359-1d49-45d5-a4cc-74ef265f4548 | Кухня
 7eb0f25a-5c99-4c3d-8f4b-3b0b227363d7 | Склад
```

Скопируйте `suuid` строки с именем вашего кухонного цеха. Название может
отличаться от буквального «Кухня»: «Горячий цех», «Холодный цех»,
«Заготовочная» — выберите тот, за которым работает нарезчик.

### Шаг 3.2. Вставить UUID в два файла

Откройте и отредактируйте эти два файла. Значения массива
`KITCHEN_STORAGE_UUIDS` должны быть **идентичны** в обоих.

**Файл 1:** [server/src/routes/orders.ts](server/src/routes/orders.ts) (ищите константу `KITCHEN_STORAGE_UUIDS`, примерно строка 29):

```typescript
const KITCHEN_STORAGE_UUIDS = [
  '123fa359-1d49-45d5-a4cc-74ef265f4548', // ← замените на UUID вашей кухни
];
```

**Файл 2:** [server/src/routes/dishes.ts](server/src/routes/dishes.ts) (аналогично, строка ~18):

```typescript
const KITCHEN_STORAGE_UUIDS = [
  '123fa359-1d49-45d5-a4cc-74ef265f4548', // ← ТОТ ЖЕ UUID
];
```

### Шаг 3.3. Несколько цехов на кухне

Если ваш нарезчик обслуживает **несколько** складов (например, «Холодный
цех» + «Горячий цех»), перечислите все UUID:

```typescript
const KITCHEN_STORAGE_UUIDS = [
  'uuid-холодного-цеха',
  'uuid-горячего-цеха',
];
```

### Шаг 3.4. Сохранить и продолжить

Сохраните оба файла. Проверка корректности UUID произойдёт на шаге 8
(после первого запуска). Если позже окажется, что UUID указан неверно —
достаточно поправить оба файла и перезапустить backend, никакие данные
не пострадают.

## 4. Применение миграций модуля к БД `arclient`

Миграции создают **15 новых таблиц** с префиксом `slicer_*` и **не трогают**
существующие таблицы вашей системы. Единственные изменения вне этого
префикса — две безопасные операции:

- `CREATE EXTENSION IF NOT EXISTS "uuid-ossp"` — если расширение уже
  установлено (а у вас оно стоит, потому что им пользуется основная KDS),
  PostgreSQL выдаст информационное предупреждение и пропустит шаг.
- Никаких `ALTER TABLE` на ваших `docm2_*` / `ctlg*` / `rgst*` —
  миграции только создают новые таблицы.

### Шаг 4.1. Резервная копия (рекомендуется)

Перед любыми изменениями продакшн-БД — снимите бэкап. Миграции безопасные
и обратимые (см. «Откат» ниже), но бэкап — это всегда дешёвая страховка.

```bash
# Linux / macOS
pg_dump -U postgres -h localhost -Fc arclient -f arclient-before-slicer.dump

# Windows (cmd/PowerShell)
"C:\Program Files\PostgreSQL\18\bin\pg_dump.exe" -U postgres -h localhost -Fc arclient -f arclient-before-slicer.dump
```

### Шаг 4.2. Запустить миграции по порядку

Миграции лежат в [server/migrations/](server/migrations/). Выполните их
**строго по номерам**. Флаг `-v ON_ERROR_STOP=1` прерывает выполнение
при первой ошибке — это важно, чтобы не продолжать после сбоя.

```bash
cd server
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/001_create_slicer_tables.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/002_seed_defaults.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/003_dish_aliases.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/004_dish_categories.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/005_dish_stoplist.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/006_kds_stoplist_sync.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/007_slicer_finished_at.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/008_dish_images.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/009_ingredient_images_filesystem.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/010_ingredient_buffer_percent.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/011_rgst3_archive_trigger.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/012_rgst3_archive_with_code.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/013_dish_priority.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/014_stop_actor_tracking.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/015_rgst3_archive_with_actor.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/016_dish_defrost.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/017_dessert_auto_park.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/018_effective_created_at.sql
psql -U postgres -h localhost -d arclient -v ON_ERROR_STOP=1 -f migrations/019_dessert_modifier_trigger.sql
```

> **⚠️ Миграция 011 создаёт BEFORE DELETE триггер на `rgst3_dishstoplist`.**
> Это **чужая** таблица основной KDS. Триггер **не меняет** её схему и
> **не блокирует** DELETE — он только перед удалением копирует данные
> в нашу `slicer_stop_history`. Без этого триггера отчёт модуля не
> увидит стопы, которые кассир поставил и снял в течение смены
> (row в rgst3 удаляется и история теряется).
>
> Согласуйте с командой основной KDS. Откат одной командой:
> ```sql
> DROP TRIGGER IF EXISTS slicer_archive_rgst3_delete_trg ON rgst3_dishstoplist;
> DROP FUNCTION IF EXISTS slicer_archive_rgst3_delete();
> ```
>
> **Миграция 012** уточняет формат: триггер пишет `target_name` с префиксом
> кода блюда (`"Д184 Дуфу острая"`). Это различает пары зал/доставка
> (одинаковое `name`, разный `code`) в отчёте. При явной связи через
> `slicer_dish_aliases` (UI «Рецепты → Связать») записи сливаются под
> именем primary-блюда и `mergeIntervals` убирает двойной учёт времени.

> **Миграция 013** — `slicer_dish_priority`: персистим per-dish приоритет
> (NORMAL / ULTRA). До неё выбор в `RecipeEditor` сбрасывался при перезагрузке.
>
> **Миграция 014** — отслеживание актора стопов: кто поставил / кто снял.
> Добавляет `stopped_by_*`, `resumed_by_*`, `actor_source` в `slicer_stop_history`
> и `slicer_dish_stoplist`; `stopped_by_*` в `slicer_ingredients`. Старые записи
> остаются с NULL (ретро-заполнения нет).
>
> **Миграция 015** — BEFORE DELETE триггер на `rgst3_dishstoplist` (переписывает
> функцию триггера из 011): дополнительно резолвит `OLD.inserter` через
> `users` и пишет `stopped_by_name/uuid` + `actor_source='kds'` в архив.
> `resumed_by_*` остаются NULL — основная KDS не хранит кто именно снял стоп.
>
> **Миграция 016** — разморозка блюд: таблица `slicer_dish_defrost` (флаг
> «требует разморозки?» per-dish, хранится на primary, alias наследует),
> колонки `defrost_started_at` / `defrost_duration_seconds` в `slicer_order_state`,
> настройки `defrost_duration_minutes` (1..60 мин, default 15) и
> `enable_defrost_sound` в `slicer_settings`. **С миграцией 020 глобальное
> `defrost_duration_minutes` удалено** — время стало per-dish.
>
> **Миграция 017** — авто-парковка десертов: колонки `dessert_category_id`
> (UUID, FK → `slicer_categories(id)` ON DELETE SET NULL),
> `dessert_auto_park_enabled` (bool, default false), `dessert_auto_park_minutes`
> (1..240, default 40) в `slicer_settings`. Seed автоматически привязывает
> к категории «Десерты» если найдётся. Правило применяется в `GET /api/orders`
> (INSERT со `status=PARKED`, `unpark_at=ordertime+X min`).
>
> **Миграция 018** — Вариант Б парковки: `effective_created_at` и
> `parked_by_auto` в `slicer_order_state`. Меняет семантику
> `accumulated_time_ms` с «активного времени до парковки» на «общее время
> парковок». Формула таймера на клиенте: `elapsed = (pivot - created_at) - accumulated_time_ms`.
> Ручная парковка возвращает заказ на историческое место в очереди (по ordertime);
> авто-парковка десерта ставит в конец очереди (как новый заказ).
>
> **Миграция 019** — уточнение правила авто-парковки десертов: паркуется только
> та позиция, у которой есть модификатор времени (по умолчанию — имя модификатора
> начинается с «Готовить» или «Ждать», настраивается в
> `slicer_settings.dessert_trigger_modifier_patterns`). Имя вида «Готовить к HH.MM»
> задаёт конкретное время возврата на сегодняшнюю дату; «Готовить позже» и
> «Ждать разъяснений» → парковка на `dessert_auto_park_minutes` минут.
>
> **Миграция 020** — per-dish время разморозки: колонка `defrost_duration_minutes`
> (INT 1..60, default 15) в `slicer_dish_defrost`. Глобальная колонка
> `slicer_settings.defrost_duration_minutes` удалена, значения скопированы в
> существующие записи `slicer_dish_defrost`. UI задания времени переехал из
> «Системные настройки» в редактор рецепта (рядом с тумблером «Требует разморозки?»).
> `enable_defrost_sound` остаётся глобальным.

Ожидаемый вывод каждой миграции — серия строк `CREATE TABLE`, `CREATE INDEX`,
`INSERT 0 N`, `ALTER TABLE` без `ERROR`.

### Шаг 4.4. Засеянные данные

- `slicer_settings` (миграция 002) — 1 строка с дефолтами: окно агрегации 5 мин, COURSE_FIFO 5 мин, `enable_defrost_sound=true`, десерты `dessert_auto_park_enabled=false` / `dessert_auto_park_minutes=40`. ## 5. Конфигурация backend (`server/.env`)

Файл `.env` **не входит в архив модуля** (он в `.gitignore` из соображений
безопасности). Создайте его вручную в папке `server/`.

### Шаг 5.1. Создать `server/.env`

```dotenv
# Подключение к БД arclient (той же, с которой работает основная KDS)
DB_HOST=localhost
DB_PORT=5432
DB_NAME=arclient
DB_USER=postgres
DB_PASSWORD=ваш_пароль_postgres

# Порт backend модуля (frontend ожидает именно 3001)
SERVER_PORT=3001
```

### Шаг 5.2. Что здесь важно

- **`DB_NAME`** должно совпадать с именем вашей продакшн-БД (обычно `arclient`).
- **`DB_USER`** должен иметь права, перечисленные в разделе 1
  (SELECT чужих таблиц + CRUD на `slicer_*`). В чужие таблицы модуль по
  умолчанию ничего не пишет. Если модуль будет использовать двустороннюю
  синхронизацию стоп-листа (раздел 10) — добавьте также `INSERT`/`DELETE`
  на `rgst3_dishstoplist`.
- **`SERVER_PORT=3001`** — не меняйте без необходимости. Фронтенд по
  умолчанию проксирует `/api/*` на `localhost:3001`. Если нужен другой
  порт — поправьте также [vite.config.ts](vite.config.ts).
- Никаких секретов в коде — всё через env. `.env` не коммитится.

### Шаг 5.3. Быстрая проверка подключения

Из папки `server/` выполните однострочник — проверит что `.env` читается
и БД отвечает:

```bash
cd server
node -e "const {Pool}=require('pg'); require('dotenv').config(); new Pool().query('SELECT current_database()').then(r=>{console.log('OK:',r.rows[0]);process.exit(0);}).catch(e=>{console.error('FAIL:',e.message);process.exit(1);});"
```

Ожидаемый вывод: `OK: { current_database: 'arclient' }`.

Если ошибка — проверьте:

- PostgreSQL запущен: `pg_isready -h localhost -p 5432`
- Пароль и имя БД в `.env` верны
- `pg_hba.conf` пускает этого пользователя с этого хоста

## 7. Запуск модуля

Модуль состоит из двух независимых процессов:

1. **Backend** (Express API, порт `3001`) — читает/пишет БД, обслуживает `/api/*`
2. **Frontend** (UI нарезчика) — отдаётся либо dev-сервером Vite на порту `3000`,
   либо статически из `dist/` после production-сборки.

### 7.1. Dev-режим (для первичной проверки)

**Терминал 1 — backend:**

```bash
cd server
npm run dev
```

Ожидаемый вывод:

```
[INFO] ts-node-dev ver. 2.0.0
[DB] Подключено к PostgreSQL: <дата>
[Server] Slicer Station API запущен на порту 3001
[Server] Эндпоинты: http://localhost:3001/api/
```

Если видите `[DB] Подключено` — backend успешно открыл соединение с
`arclient`. Если падает с `ECONNREFUSED` или `password authentication
failed` — вернитесь к шагу 5.

**Терминал 2 — frontend (из корня):**

```bash
npm run dev
```

Ожидаемый вывод:

```
VITE v6.x  ready in <ms>
➜ Local:   http://localhost:3000/
➜ Network: http://<ваш-ip>:3000/
```

Откройте в браузере: **http://localhost:3000** — это UI нарезчика.

### 7.2. Production-сборка

Когда проверили, что в dev всё работает, соберите прод-версию.

**Backend:**

```bash
cd server
npm run build      # компилирует TS → server/dist/
npm start          # запускает server/dist/index.js
```

Рекомендуется запускать как системную службу (`systemd` на Linux, Windows
Service или `pm2`) — чтобы backend поднимался вместе с сервером.

**Frontend:**

```bash
# из корня проекта
npm run build      # собирает dist/ — статика
```

Содержимое `dist/` раздавайте любым веб-сервером (nginx, caddy, IIS),
проксируя `/api/*` на backend-порт (`3001` по умолчанию). Пример для nginx:

```nginx
server {
  listen 80;
  server_name slicer.example.local;

  root /opt/kds-slicer-station/dist;
  index index.html;
  location / { try_files $uri /index.html; }

  location /api/ {
    proxy_pass http://127.0.0.1:3001/api/;
    proxy_set_header Host $host;
  }

  # Фото блюд и ингредиентов загружаются multer'ом через /api/.../:id/image
  # (POST multipart → server/public/images/{dishes,ingredients}/).
  # nginx отдаёт их напрямую с диска — быстрее, чем проксировать в Node.
  # alias указывает на ту же папку, куда пишет multer.
  location /images/ {
    alias /opt/kds-slicer-station/server/public/images/;
    expires 7d;
    add_header Cache-Control "public, immutable";
    try_files $uri =404;
  }
}
```

> **Бэкап фото.** Папки `server/public/images/dishes/` и
> `server/public/images/ingredients/` — пользовательский контент
> ресторана. Не попадают в pg_dump базы. Включите их в свой бэкап-скрипт
> отдельно (rsync, borg, и т.п.), иначе при восстановлении из бэкапа
> БД пути в `slicer_dish_images.image_path` и `slicer_ingredients.image_url`
> будут указывать на несуществующие файлы.

## 9. Начальная настройка модуля через UI нарезчика

После миграций таблицы `slicer_recipes`, `slicer_ingredients`,
`slicer_dish_aliases`, `slicer_dish_categories` — пустые. Это намеренно:
данные зависят от меню вашего ресторана и распорядка работы нарезчика,
поэтому их заполняет пользователь через UI в первый день работы.

Пока эти данные не заполнены, модуль работает, но:

- Блюда на доске KDS отображаются **без ингредиентов**.
- Все блюда уходят в хвост очереди (fallback-категория, sort_index=9999).
- Варианты блюд (`163` для зала и `Д163` для доставки) идут **двумя
  разными** карточками.

Ниже — рекомендованный порядок начальной настройки.

### Шаг 9.1. Проверить / настроить категории

**Вкладка: Админ → Категории**

5 дефолтных категорий уже созданы миграцией (VIP, Супы, Салаты, Горячее,
Десерты). При необходимости:

- Переименуйте под терминологию вашего меню.
- Добавьте свои (Стейки, Пицца, Паста, Роллы...).
- Отсортируйте drag-and-drop — именно в этом порядке блюда будут
  выстраиваться на доске KDS внутри одного временного окна.

### Шаг 9.2. Назначить каждому блюду slicer-категорию

**Вкладка: Админ → Рецепты**

Самый важный шаг на старте. При первом открытии все блюда висят в
секции «Без категории». Для каждого:

1. Нажмите «Редактировать».
2. Выберите одну или несколько категорий.
3. Сохраните.

Совет: начните с самых популярных блюд (через день-два работы в
Dashboard будет видна частотность и станет понятно, что приоритизировать).

### Шаг 9.3. Создать справочник ингредиентов

**Вкладка: Админ → Ингредиенты** (или секция в Админ-панели).

Структура — двухуровневая иерархия `родитель → дети`. Пример:

- `Картофель` (parent)
  - `Сырой картофель` (child)
  - `Пюре` (child)
  - `Фри нарезка` (child)

Ценность иерархии: стоп родителя каскадом стопит всех потомков. Когда
кончилось сырьё — одним кликом можно вывести из обращения все производные.

Поля ингредиента:

- Название
- Родитель (опционально — если это child)
- Единица измерения (граммы, штуки, литры, ...)
- Изображение (опционально)

### Шаг 9.4. Составить рецепты

**Вкладка: Админ → Рецепты → выбрать блюдо → «Редактировать»**

Для каждого блюда — список ингредиентов с граммовкой на одну порцию:

```
Цезарь с курицей:
  - Куриное филе     80 г
  - Айсберг           50 г
  - Соус цезарь       20 г
  - Сухарики          15 г
  - Пармезан          10 г
```

Граммовка используется не только для подсказки нарезчику, но и для
расчёта расхода ингредиентов в Dashboard.

### Шаг 9.5. Связать алиасы блюд

**Вкладка: Админ → Рецепты → карточка блюда → кнопка «Связать»**

Если в меню одно логическое блюдо имеет несколько вариантов (`163
Баклажаны` для зала, `Д163 Баклажаны` для доставки) — свяжите алиас
с основным блюдом. После этого:

- На доске нарезчика они показываются как одно блюдо (агрегируются).
- Рецепт ведётся только на основное, алиас наследует.
- Категории основного автоматически копируются на алиас.

Ограничение: у одного alias может быть только один primary (PK на уровне БД).

### Шаг 9.6. Проверить системные настройки

**Вкладка: Админ → Настройки**

Основные параметры:

- **Окно агрегации** — сколько минут заказы одного стола объединяются
  в один стек. Рекомендация: 5 минут.
- **Окно COURSE_FIFO** — размер временного окна для гибридной сортировки.
  Рекомендация: 300 секунд.
- **Удержание истории** — сколько минут показывать завершённые заказы
  в «Недавних». Рекомендация: 60 минут.
- **Активные правила сортировки** — по умолчанию `['ULTRA', 'COURSE_FIFO']`.

### Шаг 9.7. Боевой тест

Сделайте один тестовый заказ на кассе и проследите:

1. Заказ появился на доске KDS нарезчика в течение 4 секунд ✓
2. Видны ингредиенты из рецепта ✓
3. Блюдо в правильной категории (не в «Без категории») ✓
4. Нажатие «Готово» убирает заказ и сохраняет в историю ✓

Если все 4 пункта — модуль готов к боевой работе.

## 10. Двусторонняя синхронизация стоп-листа (опционально)

### Что это и зачем

По умолчанию модуль нарезчика работает **в одностороннем режиме**:
- ✅ Стоп блюда из основной KDS (`rgst3_dishstoplist`) **виден нарезчику** (мы читаем эту таблицу через UNION в `GET /api/dishes`)
- ❌ Стоп блюда от нарезчика (или каскадный стоп от стопнутого ингредиента) **не виден** в основной KDS — кассиры могут продолжать пробивать блюдо, даже если у нарезчика кончились ингредиенты

**Двусторонняя синхронизация** решает вторую проблему. Когда нарезчик ставит блюдо на стоп (вручную или каскадом), модуль автоматически вставляет соответствующую строку в `rgst3_dishstoplist` — кассовая система сразу видит это и блокирует блюдо в UI кассы. При снятии стопа модуль удаляет ровно ту строку, которую сам создал.

### Безопасность по умолчанию

Синхронизация **выключена из коробки** (`slicer_settings.enable_kds_stoplist_sync = false`). Без явных действий программистов заказчика модуль ничего не пишет в `rgst3_dishstoplist`. Это сделано чтобы **не сломать работающую систему** — включение требует осознанного решения и заполнения конфига.

### Что модуль будет делать при включении

1. **При стопе блюда** (ручном или каскадном) — `INSERT INTO rgst3_dishstoplist` с обязательными полями:
   - `rgst3_ctlg15_uuid__dish` — id блюда
   - `rgst3_ctlg11_uuid__restaurant` — из конфига
   - `rgst3_ctlg14_uuid__shift` — текущая открытая смена (динамически)
   - `rgst3_ctlg16_uuid__restaurantmenu` — из конфига
   - `rgst3_ctlg5_uuid__responsible` — из конфига
   - `inserter` / `updater` — текстовый идентификатор из конфига
   - `comment` — причина стопа (например «Missing: Картофель» или ручной текст)
2. **При снятии стопа** — `DELETE FROM rgst3_dishstoplist WHERE suuid = ?` по сохранённому suuid (модуль удаляет только свои строки, чужие не трогает)
3. **Всё в одной транзакции** с `slicer_dish_stoplist`. Если запись в `rgst3_dishstoplist` упадёт (нет прав, неверные UUID, конфликт схемы) — транзакция откатывается полностью, состояние модуля и кассы остаётся консистентным.

### Возможные ошибки при включённой синхронизации

| Ошибка в UI или логах backend | Причина | Решение |
|---|---|---|
| `enable_kds_stoplist_sync = TRUE, но slicer_kds_sync_config пуст` | Включили флаг, но не заполнили конфиг | Выполните Шаг 3 |
| `Нет открытой смены в ctlg14_shifts` | В системе нет открытой смены | Откройте смену в основной KDS перед использованием стоп-листа |

## 11. Замер времени готовки повара (опционально)

### Зачем

Процесс приготовления блюда выглядит так:

```
Заказ на кассе
    ↓
Нарезчик подготовил ингредиенты  →  жмёт «Готово» (slicer_order_state.finished_at = NOW())
    ↓
Повар готовит блюдо
    ↓
Раздача/пасс отмечают готово     →  ваша система пишет docm2tabl1_cooktime = NOW()
    ↓
Официант забирает блюдо
```

Промежуток `docm2tabl1_cooktime − slicer_order_state.finished_at` — это и есть
**чистое время готовки повара** (между передачей от нарезчика и выдачей на
раздачу). Очень полезная метрика: даёт средний темп повара, помогает находить
узкие места, позволяет сравнивать повара/блюда/смены.

### Где смотреть в UI

Метрика встроена в модуль. В UI нарезчика откройте вкладку **Отчёты**,
выберите период и нажмите **OK** — на экране появится таблица
**«Скорость готовки повара»** (фиолетовый акцент) рядом со «Скоростью
отдачи». Интерфейс аналогичен: группировка по категории → блюдо,
колонки «Название / Кол-во / Ср. Время», поиск по названию, сортировка
по любой колонке.

Данные таблица берёт из endpoint `GET /api/history/dashboard/chef-cooking-speed`,
который делает JOIN `slicer_order_state` ↔ `docm2tabl1_items` и отдаёт
пары `(finished_at, docm2tabl1_cooktime)` за выбранный период. Алиасы
блюд резолвятся автоматически (варианты `163`/`Д163` агрегируются как
одно блюдо).

### Предусловие: ваша система должна писать `docm2tabl1_cooktime`

Модуль **не** пишет в это поле (начиная с миграции 007). Замер работает
только если ваша основная KDS / раздача пишет в `docm2tabl1_cooktime` в
момент, когда отмечает блюдо как приготовленное.

Проверка — выполните на живой БД:

```sql
SELECT count(*) AS total,
       count(docm2tabl1_cooktime) AS with_cooktime,
       count(*) FILTER (WHERE docm2tabl1_cooked = true) AS cooked_true
  FROM docm2tabl1_items
  WHERE docm2tabl1_ordertime > NOW() - INTERVAL '7 days';
```

Если `with_cooktime` ≈ `cooked_true` — ваша система пишет метку, всё готово.
Если `with_cooktime = 0` а `cooked_true > 0` — поле не заполняется, нужно
договориться с командой основной KDS, чтобы начали писать, или завести
триггер (см. ниже).

### SQL для разовых выгрузок / проверки

Для UI-отчёта ничего выполнять не нужно — он работает сам. Этот SQL
нужен если хотите выгрузить метрику во внешний BI / Excel / кассовый
отчёт, либо убедиться что данные собираются:

```sql
-- Среднее время готовки повара (в минутах) по блюдам за сегодня
SELECT dish.name AS dish_name,
       count(*) AS orders,
       round(avg(extract(epoch FROM items.docm2tabl1_cooktime - state.finished_at)) / 60, 2) AS avg_cook_minutes,
       round(min(extract(epoch FROM items.docm2tabl1_cooktime - state.finished_at)) / 60, 2) AS min_minutes,
       round(max(extract(epoch FROM items.docm2tabl1_cooktime - state.finished_at)) / 60, 2) AS max_minutes
  FROM slicer_order_state state
  JOIN docm2tabl1_items items ON items.suuid::text = state.order_item_id
  JOIN ctlg15_dishes    dish  ON dish.suuid = items.docm2tabl1_ctlg15_uuid__dish
 WHERE state.finished_at IS NOT NULL
   AND items.docm2tabl1_cooktime IS NOT NULL
   AND items.docm2tabl1_cooktime > state.finished_at   -- защита от кривых данных
   AND state.finished_at >= CURRENT_DATE
 GROUP BY dish.name
 ORDER BY avg_cook_minutes DESC
 LIMIT 20;
```

Такой же запрос с группировкой по смене (`docm2_orders` → `ctlg14_shifts`)
даёт метрику «среднее время готовки за смену», с группировкой по повару —
«среднее время по повару» (если у вас в `docm2tabl1_items` хранится повар).

### Если `docm2tabl1_cooktime` у вас не пишет никто

Добавьте триггер, который будет заполнять поле в момент перехода
`docm2tabl1_cooked` в `true`:

```sql
CREATE OR REPLACE FUNCTION slicer_fill_cooktime() RETURNS trigger AS $$
BEGIN
  IF NEW.docm2tabl1_cooked = true
     AND (OLD.docm2tabl1_cooked IS DISTINCT FROM true)
     AND NEW.docm2tabl1_cooktime IS NULL THEN
    NEW.docm2tabl1_cooktime := NOW();
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER slicer_fill_cooktime_trg
  BEFORE UPDATE ON docm2tabl1_items
  FOR EACH ROW
  EXECUTE FUNCTION slicer_fill_cooktime();
```

⚠️ Это единственное место, где этот модуль предлагает создать триггер на
вашу таблицу. Решение о применении — **на вашей стороне**, потому что это
меняет поведение вашей `docm2tabl1_items`. Если у вас уже есть другой
механизм проставления `cooktime` — пропускайте этот шаг.

## 12. Возможные проблемы

### Проблема: экран KDS пустой, хотя заказы на кассе есть

**Симптом.** Backend работает, `/api/health` ок, но `/api/orders`
возвращает `[]` и доска пустая.

**Причины (в порядке вероятности):**

1. **`KITCHEN_STORAGE_UUIDS` не совпадает с реальным UUID цеха** — фильтр
   отрезает все позиции. Вернитесь к разделу 3. Это **самая частая**
   проблема при переносе и она **молчаливая** (в логах чисто).
2. **Нет открытой смены** в `ctlg14_shifts`:
   ```sql
   SELECT suuid, ctlg14_closed FROM ctlg14_shifts
     WHERE ctlg14_closed = false OR ctlg14_closed IS NULL;
   ```
   Если пусто — откройте смену в основной системе.
3. **Статус позиции не ACTIVE.** Backend фильтрует по
   `COALESCE(state.status, 'ACTIVE') NOT IN ('COMPLETED', 'CANCELLED')`.

### Проблема: все блюда в «Без категории»

**Симптом.** На вкладке «Админ → Рецепты» все блюда в секции «Без
категории»; на доске KDS блюда уходят в хвост очереди.

**Причина.** Таблица `slicer_dish_categories` пустая — это **нормальное**
состояние после установки модуля. Данные о категориях в вашей
`ctlg15_dishes.ctlg15_ctlg38_uuid__goodcategory` не мапятся напрямую
на slicer-категории.

**Решение.** Шаг 9.2.

### Проблема: заказы приходят, но без ингредиентов

**Симптом.** Карточки на доске отображаются с пустым списком ингредиентов.

**Причина.** Таблица `slicer_recipes` пустая.

**Решение.** Шаг 9.4.

### Проблема: `163 Баклажаны` и `Д163 Баклажаны` — два разных блюда

**Симптом.** Варианты одного блюда (зал / доставка) идут двумя карточками.

**Причина.** В `slicer_dish_aliases` нет связи.

**Решение.** Шаг 9.5.

### Проблема: «расширение uuid-ossp уже существует» при миграции 001

**Симптом:** при выполнении `001_create_slicer_tables.sql`:
```
ЗАМЕЧАНИЕ: расширение "uuid-ossp" уже существует, пропускается
```

**Статус.** **Безопасно, игнорируйте.** Миграция использует
`CREATE EXTENSION IF NOT EXISTS` — расширение не трогается, если уже есть.

### Проблема: `ECONNREFUSED` в backend при старте

**Симптом.** `npm run dev` в `server/` падает с ошибкой подключения к PostgreSQL.

**Причины:**

- PostgreSQL не запущен → `pg_isready -h localhost -p 5432`.
- Неправильный порт в `.env` (`DB_PORT`).
- Firewall блокирует localhost:5432 (редко).

### Проблема: `password authentication failed for user "postgres"`

**Симптом.** Backend подключается, но сразу получает отказ.

**Решение:**

- Проверьте `DB_PASSWORD` в `server/.env`.
- Проверьте `pg_hba.conf` — возможно требуется `md5` / `scram-sha-256`
  вместо `trust` для локальных подключений.
- Убедитесь, что пользователь имеет доступ к БД:
  `GRANT CONNECT ON DATABASE arclient TO <имя_пользователя>;`

### Проблема: в логах backend `permission denied for table rgst3_dishstoplist`

**Симптом.** Ошибка возникает только при включённой двусторонней
синхронизации (раздел 10).

**Решение.** Раздел 10, шаг 4 — дайте пользователю БД права на INSERT/DELETE.

### Проблема: Dashboard показывает белый экран

**Симптом.** Вкладка Dashboard не отображается / ошибка в UI.

**Причина.** Обычно — пустая `slicer_order_history` (ещё не было
завершённых заказов).

**Временный обходной путь.** Поработать с модулем хотя бы день, после
первых завершений история наполнится.
