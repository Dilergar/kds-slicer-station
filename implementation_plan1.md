# Smart Aggregation — Волновая система с симуляцией очереди

## Проблема

Текущая "Smart Aggregation (By Table)" просто объединяет одинаковые блюда с одного стола. Настоящая Smart Aggregation должна:

1. **Определять "волну"** обслуживания каждого стола (какую категорию блюд готовить следующей)
2. **Симулировать всю очередь** — заранее рассчитать порядок, предполагая что блюда готовятся сверху вниз
3. **Агрегировать** одинаковые блюда из РАЗНЫХ столов (когда они попадают на одну позицию)
4. **Один плоский список** — нарезчик просто берёт первое, без секций и разделителей

> **Главный принцип:** Нарезчик не думает — он просто берёт первый заказ из списка и это всегда самое правильное решение. Список заранее просчитан от начала до конца.

## Пример работы

### Исходные заказы:
```
Стол №1: Суп борщ, Салат цезарь, Стейк
Стол №2: Салат цезарь, Тирамису
Стол №3: Стейк
Стол №4: Суп борщ, Лазанья, Творог сладкий
Стол №5: Тирамису
```

### Результат — один готовый список:
```
1. Суп борщ × 2           [Столы №1, №4]
2. Салат цезарь × 2        [Столы №1, №2]    ← объединены!
3. Стейк × 2              [Столы №1, №3]     ← объединены!
4. Лазанья × 1            [Стол №4]
5. Тирамису × 2           [Столы №2, №5]     ← объединены!
6. Творог сладкий × 1     [Стол №4]
```

### Почему именно так:
1. **Позиция #1**: Волны столов: №1=Суп, №2=Салат, №3=Горячее, №4=Суп, №5=Десерт. Самая приоритетная категория = Суп (sort_index=0). Борщ для столов №1 и №4 → объединяем.
2. **Позиция #2**: Симулируем что борщ готов → стол №1 волна=Салат, стол №4 волна=Горячее. Теперь столы №1 и №2 оба ждут салат → объединяем Цезарь × 2.
3. **Позиция #3**: Симулируем что цезарь готов → стол №1 волна=Горячее, стол №2 волна=Десерт. Столы №1 и №3 оба ждут горячее → объединяем Стейк × 2.
4. **И так далее...**

## Алгоритм `buildSmartQueue()`

```
Вход: orders[], orderHistory[], dishes[], categories[]

1. Развернуть все стековые заказы (quantity_stack + table_stack) 
   в плоский массив: [{dishId, tableNumber, orderId, orderedAt}, ...]

2. Из orderHistory определить уже поданные категории для каждого стола

3. ЦИКЛ СИМУЛЯЦИИ (пока есть неразмещённые блюда):
   a. Для каждого стола вычислить текущую волну 
      (= первая неподанная категория по sort_index, у которой есть pending-блюда)
   b. Собрать все блюда, чья категория совпадает с волной их стола
   c. Из них выбрать самую приоритетную категорию (наименьший sort_index)
   d. Сгруппировать одинаковые блюда этой категории → одна позиция
   e. Добавить позицию в очередь
   f. СИМУЛЯЦИЯ: пометить эти блюда как "поданные" 
      (обновить served-категории для этих столов)
   g. Повторить с шага (a)

4. Вернуть готовый список SmartQueueGroup[]
```

## Ключевое архитектурное решение: Стеки vs Плоская модель

Текущая модель Order использует `quantity_stack` + `table_stack` (стековая).
Алгоритм разворачивает стеки в плоский массив для симуляции, а результат возвращает как агрегированные группы. Виртуальные Order-объекты создаются из SmartQueueGroup для совместимости с текущим OrderCard.

## Решения по ключевым вопросам

| # | Вопрос | Решение |
|---|--------|---------|
| 1 | Smart Aggregation заменяет текущую? | **Да** — старая "(By Table)" удаляется |
| 2 | Секции "в волне / ожидает"? | **Нет** — один плоский список, всё заранее рассчитано |
| 3 | Time-Window vs Smart | **Взаимоисключающие** — при включении одного другой OFF |
| 4 | Несколько нарезчиков? | Part Done уже решает — каждый отмечает сколько сделал |
| 5 | Паркованные заказы? | **Исключаются** из расчёта волн |
