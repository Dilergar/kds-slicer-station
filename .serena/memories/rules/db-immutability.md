# 🔒 ПРАВИЛО: Неприкосновенность чужой БД

**КРИТИЧЕСКИ ВАЖНО.** БД `arclient` принадлежит IT-команде заказчика. Это рабочая продакшн-схема, которую они передали для разработки модуля Slicer.

## ❌ ЗАПРЕЩЕНО изменять существующие таблицы KDS

Списки таблиц которые **НЕЛЬЗЯ ТРОГАТЬ**:
- `docm1_*` — `docm9_*` (документы: заказы, калькуляции, закупки и т.д.)
- `ctlg1_*` — `ctlg44_*` (справочники: блюда, столы, смены, категории, принтеры, сотрудники и т.д.)
- `rgst1_*` — `rgst4_*` (регистры: стоп-листы, бронь и т.д.)
- `objects`, `points`, `sessions`, `users`, `interfaces`, `settings` (системные)
- Любые другие таблицы без префикса `slicer_`

### Конкретно нельзя:
- Добавлять/удалять/переименовывать колонки
- Менять типы данных, constraints, FK, индексы
- Создавать триггеры/функции/views которые модифицируют эти таблицы
- Делать миграции меняющие чужую схему

## ✅ РАЗРЕШЕНО
- `SELECT` из любых существующих таблиц
- Создавать новые таблицы **только с префиксом `slicer_`**
- Создавать свои индексы только на `slicer_*` таблицах
- **Единственное опциональное исключение для записи в чужие таблицы:** `INSERT/DELETE` в `rgst3_dishstoplist` через адаптер `server/src/services/kdsStoplistSync.ts`, только при включённом `slicer_settings.enable_kds_stoplist_sync = true` (OFF по умолчанию). См. `Инструкция.md` раздел 10.

## ❌ БОЛЬШЕ НЕ ПИШЕМ: `docm2tabl1_items.docm2tabl1_cooked`
Убрано в миграции 007 (2026-04-18). Нарезчик закрывает позицию только в `slicer_order_state.status='COMPLETED'` + `slicer_order_state.finished_at=NOW()`. Поле `docm2tabl1_cooked`/`docm2tabl1_cooktime` — под управлением основной KDS. Разница `docm2tabl1_cooktime - slicer_order_state.finished_at` = время готовки повара (отчёт в `Инструкция.md` раздел 11).

## ⚠️ ТРИГГЕР НА ЧУЖОЙ ТАБЛИЦЕ (миграция 011, 2026-04-19)
`BEFORE DELETE` на `rgst3_dishstoplist` → `slicer_archive_rgst3_delete()` копирует OLD row в `slicer_stop_history`. **Не** меняет схему rgst3, **не** блокирует DELETE, **не** пишет в чужие таблицы. Нужно для полноты Dashboard-отчёта: без триггера снятия стопа кассиром в течение смены теряются (rgst3 DELETE безвозвратный). Защита от дубликатов при двусторонней синхронизации: пропускает DELETE где `OLD.inserter = slicer_kds_sync_config.inserter_text`. Откат `DROP TRIGGER + DROP FUNCTION` — следов в чужой схеме 0.

## Цель правила
Когда модуль передаётся заказчику:
1. Программисты получают SQL из `server/migrations/`
2. Выполняют на своей продакшн-БД
3. **Всё сразу работает** без ручных правок схемы
4. Никаких конфликтов с их рабочей системой

## Как правильно расширять функциональность

Если нужна новая логика/данные:
1. Создай **новую** таблицу `slicer_*`
2. Свяжи с чужими таблицами через **теневой подход** — VARCHAR-ссылка на UUID без FK. Пример: `slicer_order_state.order_item_id VARCHAR` → `docm2tabl1_items.suuid`
3. JOIN на уровне backend при выдаче клиенту
4. Задокументируй в `BD_docs/tables/slicer_*.md`

## Эталонный пример: slicer_order_state
Хранит парковку, статус, merge-состояние заказов нарезчика. Связывается с `docm2tabl1_items` через `order_item_id` (VARCHAR). Исходная таблица `docm2tabl1_items` не меняется ни на байт.

## Эталонная миграция
`server/migrations/001_create_slicer_tables.sql` — только CREATE TABLE для `slicer_*`, ни одного ALTER для чужих таблиц. Безопасно для продакшна.
