# Фильтр цехов (Storage Filter) — Whitelist

## Задача
Нарезчик должен видеть только кухонные блюда. Всё остальное (бар, хозка, битые ссылки на удалённые склады) — скрыто. Новые блюда добавленные в меню должны появляться автоматически.

## Подход: Whitelist + Dual-source
Показываем только позиции/блюда с UUID кухонного склада. Для справочника блюд используем два источника чтобы новые блюда появлялись сразу.

## Источник правды
Таблица `ctlg17_storages`. Реальные склады ресторана Жарокова (проверено на прод-дампе):
- `49aeb05c-966b-4d42-bf80-595de514122a` — **Кухня** ✅ (whitelist)
- `48dded95-9c6d-421e-9d7e-58e49a3e7889` — Бар ❌
- `94cb93cc-0bf8-41ae-b255-131d91a15017` — Склад ❌

## Реализация

### 1. server/src/routes/orders.ts — GET /api/orders
```typescript
const KITCHEN_STORAGE_UUIDS = [
  '49aeb05c-966b-4d42-bf80-595de514122a', // Кухня Жарокова
];
```
```sql
AND items.docm2tabl1_ctlg17_uuid__storage IN ($N, ...)
```

### 2. server/src/routes/dishes.ts — GET /api/dishes
Та же константа. Блюдо попадает в список если **хотя бы одно** условие:
```sql
-- Источник 1: меню ресторана (новые блюда появляются сразу)
EXISTS (
  SELECT 1 FROM ctlg18_menuitems mi
  WHERE mi.ctlg18_ctlg15_uuid__dish = d.suuid
    AND mi.isfolder = false
    AND mi.ctlg18_ctlg17_uuid__storage IN ($N, ...)
)
-- Источник 2: историческая выборка (fallback для старых блюд)
OR EXISTS (
  SELECT 1 FROM docm2tabl1_items i
  WHERE i.docm2tabl1_ctlg15_uuid__dish = d.suuid
    AND i.docm2tabl1_ctlg17_uuid__storage IN ($N, ...)
)
```

## Почему два источника для /api/dishes
- **ctlg18_menuitems** — пункты меню ресторана. Как только новое блюдо добавляют в меню с кухонным складом, оно СРАЗУ появляется в редакторе рецептов нарезчика, даже без первого заказа.
- **docm2tabl1_items** — историческая выборка. Fallback для блюд которые ещё не занесены в новое меню, но уже заказывались.

Логика OR гарантирует что:
- Новые блюда: появляются через ctlg18_menuitems
- Старые блюда: попадают через docm2tabl1_items
- Блюда из бара/хозки: отбрасываются обоими источниками

## Где фильтр применён
Фильтр в **двух источниках правды**:
1. `GET /api/orders` — KDS-доска
2. `GET /api/dishes` — справочник блюд для Рецептов, админки, KPI

Всё остальное защищено транзитивно: фронт работает только с тем что пришло из этих endpoint'ов, история/dashboard заполняется только завершёнными кухонными заказами.

## Битые ссылки
Часть позиций ссылается на удалённые склады (UUID не существует в ctlg17_storages). При whitelist они автоматически отбрасываются. Их поправят вручную в БД позже.

## При деплое на другой ресторан
1. Найти UUID кухонного склада в `ctlg17_storages` целевой БД:
   ```sql
   SELECT suuid, name FROM ctlg17_storages ORDER BY name;
   ```
2. Обновить `KITCHEN_STORAGE_UUIDS` в обоих файлах (`orders.ts` и `dishes.ts`)
3. В будущем можно вынести в `slicer_settings` как JSONB массив

**⚠️ Молчаливый баг при переносе:** если UUID не совпадает, экран KDS остаётся пустым без единой ошибки в логах. Описано в `Инструкция.md` первым пунктом после требований.
