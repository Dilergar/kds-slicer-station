# Миграция 024: шаг курса по умолчанию 600 сек + семантика «окно уступки»

## Дата выполнения
2026-07-11

## Файл
`server/migrations/024_course_pace_default_600.sql`

## Что делает
1. `ALTER COLUMN course_pace_seconds SET DEFAULT 600` (было 120).
2. Условный `UPDATE`: если в строке настроек стоит прежний дефолт 120 — меняет на 600. Вручную выставленное значение (≠120) не перезаписывается.
3. Обновляет `COMMENT ON COLUMN` под новую семантику.

**Структура таблиц не меняется** — только DEFAULT, значение и комментарий колонки, добавленной миграцией 023.

## Зачем
По итогам ревью очереди (2026-07-11) владелец зафиксировал целевую семантику умной очереди. Симуляции на реальном коде показали два расхождения со старым поведением:

### 1. «Готово» пересобирало очередь (промоушен курса)
Номер курса гостя считался только по позициям, **оставшимся** на доске. После отдачи супа горячее того же стола становилось «курсом 0», его виртуальное время падало до времени пробития стола — и стол получал все свои блюда подряд, обгоняя первые курсы гостей, пришедших позже. Побочный эффект: настройка шага курса практически не влияла на живой порядок (симуляция со 120с и 600с давала идентичный результат).

**Исправление — «замороженные» курсы (только код, БД не меняется):**
- `GET /api/orders` дополнительно отдаёт каждому заказу контекст визита его гостя:
  - `visit_completed_dish_ids` — канонические `dish_id` позиций, уже отданных нарезчиком (строки `slicer_order_state.status = 'COMPLETED'`, JOIN по **открытым** чекам: `docm2_orders.docm2_closed = false`; чек закрыт → визит окончен, новая компания за столом начинает с чистых курсов);
  - `visit_started_at` — самое раннее время пробития среди отданных позиций.
- `smartQueue.buildSmartQueue` считает курсы по **всем** позициям визита: активные + активно размораживающиеся + отданные. Виртуальное время позиции неизменно всю её жизнь: «Готово» просто снимает верхнюю карточку, остальные не двигаются.

### 2. Дефолт 120 сек не соответствовал целевой очереди
Эталонный пример владельца (стол 1: суп+салат+бифштекс; стол 2: суп+бифштекс; стол 3: бифштекс; стол 4: десерт → очередь `Суп×2(1,2) → Бифштекс×2(2,3) → Десерт(4) → Салат(1) → Бифштекс(1)`) воспроизводится движком при шаге **600 сек** и ломается при 120/0 сек (салат стола 1 обгоняет бифштексы и десерт). Проверено симуляцией на реальном коде.

## Новая семантика `course_pace_seconds` — «окно уступки»
На сколько секунд курс N стола уступает дорогу первым курсам гостей, пришедших позже: `vt = старт визита + номер_курса × шаг`. До наступления `vt` первые блюда новых столов проходят вперёд; после — позицию уже никто не обгонит (любой новый заказ получает `vt` не раньше своего пробития). **На скорость нарезки не влияет**: при свободной очереди блюдо режется сразу — это только порядок при конкуренции.

Замеры перегрузки (20 супов-новичков каждые 30с, нарезка 45с/карточка): при шаге 2 мин салат/бифштекс стола 1 уступают 4/8 новичкам, при 5 мин — 10/20, при 10 мин — всем 20 (поток кончается раньше окна). Средн. ожидание новичков при этом улучшается: 4.4 → 3.4 мин.

## Изменения в коде
- `server/src/routes/orders.ts` — GET: запрос контекста визита (`slicer_order_state COMPLETED` × открытые чеки) + поля `visit_completed_dish_ids` / `visit_started_at` в ответе.
- `smartQueue.ts` — `buildSmartQueue`: курсы/старт гостя считаются по визиту (активные + размораживающиеся + отданные); хелперы `orderWaveKeys`, `courseCategoryOf`.
- `types.ts` — поля `Order.visit_completed_dish_ids` / `visit_started_at`; комментарий `SystemSettings.coursePaceSeconds`.
- `components/SlicerStation.tsx` — фолбэк шага `|| 600`.
- `components/admin/SystemSettingsTab.tsx` — заголовок «⏱️ Шаг курса — окно уступки», новое описание, дефолты 600.
- `App.tsx` — инертный начальный дефолт 600.

## Откат
```sql
ALTER TABLE slicer_settings ALTER COLUMN course_pace_seconds SET DEFAULT 120;
UPDATE slicer_settings SET course_pace_seconds = 120 WHERE id = 1 AND course_pace_seconds = 600;
```
Код при этом продолжит работать (значение читается из БД), но целевая очередь владельца потребует вернуть 600.
