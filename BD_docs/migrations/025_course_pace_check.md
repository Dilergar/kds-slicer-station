# Миграция 025: CHECK-ограничение на шаг курса (course_pace_seconds)

## Дата выполнения
2026-07-11

## Файл
`server/migrations/025_course_pace_check.sql`

## Что делает
1. Клампит текущее значение `course_pace_seconds` в диапазон 10..3600, если
   оно вне границ (защита от падения `ADD CONSTRAINT` на чужой БД).
2. `ADD CONSTRAINT slicer_settings_course_pace_seconds_valid CHECK
   (course_pace_seconds BETWEEN 10 AND 3600)` (через `DROP CONSTRAINT IF
   EXISTS` — миграция идемпотентна, повторный запуск безопасен).

**Структура таблиц не меняется** — только ограничение на колонку, добавленную
миграцией 023.

## Зачем
По итогам полного ревью 2026-07-11: границы 10..3600 проверялись только в API
(`PUT /api/settings`) и в UI-инпуте, но не в схеме. Прямой SQL (или будущий
путь записи мимо валидации) мог записать 0 или отрицательное значение — тогда
виртуальное время позиций схлопывается (`vt = старт визита + курс × 0`),
умная очередь молча деградирует к чистому FIFO и эталонная очередь владельца
ломается (см. миграцию 024). Схема — единственный слой, который программисты
заказчика гарантированно выполняют при деплое, поэтому инвариант закреплён в
ней (симметрично `CHECK (dessert_auto_park_minutes BETWEEN 1 AND 240)` из
миграции 017).

## Связанные изменения в коде (той же датой)
- `server/src/routes/settings.ts` — валидация диапазонов вынесена в общий
  хелпер `validateIntRange` (границы совпадают с CHECK'ами БД).
- `smartQueue.ts` — параметр `coursePaceMs` функции `buildSmartQueue` стал
  обязательным (раньше имел вводящий в заблуждение дефолт 120000 мс,
  противоречащий дефолту БД 600 сек).

## Откат
```sql
ALTER TABLE slicer_settings
  DROP CONSTRAINT IF EXISTS slicer_settings_course_pace_seconds_valid;
```
