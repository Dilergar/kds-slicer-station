# Миграция 015 — триггер архивации пишет актора стопа

**Дата:** 2026-04-23
**Файл:** `server/migrations/015_rgst3_archive_with_actor.sql`

## Зачем

Дополнение к миграции 014. Когда кассир снимает стоп через основную KDS, триггер `slicer_archive_rgst3_delete_trg` (из миграции 011, обновлён в 012) архивирует удаляемую строку в `slicer_stop_history`. До миграции 015 триггер не заполнял новые `stopped_by_*` колонки — получалось что для kds-стопов автора не было видно.

## Что меняется

Функция `slicer_archive_rgst3_delete()` теперь:

1. Читает `OLD.inserter` (UUID юзера как text).
2. `SELECT uuid, login FROM users WHERE uuid::text = OLD.inserter`.
3. Если нашёлся — пишет `stopped_by_uuid` + `stopped_by_name` в новую строку `slicer_stop_history`.
4. Если не нашёлся (исторические данные с другим источником inserter) — пишет `Unknown (<первые-8-символов-inserter>)` в name, uuid = NULL.
5. `actor_source = 'kds'` всегда.

## Что НЕ меняется

`resumed_by_*` остаются NULL для kds-архивированных записей. Причина:

- `rgst3_dishstoplist` не хранит «кто удалил row».
- `OLD.updater` часто равен `OLD.inserter` (основная KDS не обновляет стоп-лист между INSERT и DELETE). Извлечь «сняшего» оттуда нельзя.

В UI в колонке «Снял» для kds-стопов отображается «—».

## Триггер — что важно

- BEFORE DELETE, не блокирует удаление.
- Чужая схема `rgst3_dishstoplist` не трогается (структура неизменна).
- Защита от дубликатов при включённой двусторонней синхронизации (миграция 011): если `OLD.inserter` совпадает с нашим `slicer_kds_sync_config.inserter_text`, триггер пропускает запись (мы её уже записали сами).

## Откат

Если нужно откатиться — применить SQL из миграции 012 (версия без actor), триггер перезапишется. Новые колонки из миграции 014 остаются как есть (NULLable), это не ломает обратную совместимость.
